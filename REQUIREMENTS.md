# BK P2P Aggregator — 要件定義書

## プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | BK P2P Aggregator（BK P2P統合ダッシュボード） |
| 目的 | 複数P2P取引所のレート・流動性を一画面で監視し、条件に基づく自動取引を実行する |
| ユーザー | 社内スタッフ（権限分けあり） |
| プラットフォーム | Web（ブラウザ）+ Telegram Bot |
| 通貨ペア | JPY ↔ 仮想通貨（USDT/BTC/ETH等） |

#### 2.1.2 表示する情報

| カテゴリ | 項目 | 説明 |
|---------|------|------|

| | 売値（Bid） | 仮想通貨をJPYで売るときの価格 |
| | スプレッド | 買値と売値の差 |
| | 中央値 | 市場の基準レート |
| | 乖離率 | 市場レートからの乖離率(%) |
| **流動性** | 在庫量 | マーチャントが提供可能な総額 |
| | 注文数 | アクティブな注文件数 |
| | 板の深さ | 価格帯ごとの流動性分布 |
| **マーチャント** | 完了率 | 取引完了率(%) |
| | 取引回数 | 累計取引件数 |
| | 評価 | ユーザー評価 |
| | 平均応答時間 | 注文から応答までの時間 |
| | オンライン状態 | 現在オンラインか |
| **決済方法** | 銀行振込 | 対応する銀行一覧 |
| | その他 | PayPay、LINE Pay等 |
| **フィルター** | 金額範囲 | 取引したい金額でフィルタ |

### 2.2 取引機能

#### 2.2.1 ワンクリック取引

```
ダッシュボードの [買う] / [売る] ボタンから:

1. 取引所を選択（またはランキング1位を自動選択）
2. 金額を入力
3. 確認画面
4. 実行

→ 対象取引所のP2P APIで注文を発行
→ エスクロー開始
→ 銀行振込の実行（手動 or 指示）
→ 取引完了の確認
→ ログに記録
```

#### 2.2.2 条件設定型自動取引（ルールエンジン）

```
【ルール設定画面】

ルール名: [                    ]

条件（AND/OR組み合わせ可能）:
  ├── レート条件
  │   ├── USDT買値が [¥___] 以下になったら
  │   ├── USDT売値が [¥___] 以上になったら
  │   ├── スプレッドが [¥___] 以上になったら
  │   └── 乖離率が [___]% 以上になったら
  │
  ├── 取引所条件
  │   ├── 対象取引所: [全て / 選択]
  │   ├── マーチャント完了率: [___]% 以上
  │   └── マーチャント取引回数: [___] 回以上
  │
  ├── 時間条件
  │   ├── 実行時間帯: [HH:MM] 〜 [HH:MM]
  │   ├── 曜日: [月火水木金土日]
  │   └── 有効期限: [YYYY-MM-DD] まで
  │
  └── 流動性条件
      └── 在庫が [¥___] 以上ある場合のみ

アクション:
  ├── 取引タイプ: [買い / 売り]
  ├── 通貨: [USDT / BTC / ETH]
  ├── 金額: [¥___] （可変設定可能）
  ├── 上限: 1日あたり [¥___] まで（可変）
  ├── 1回あたり上限: [¥___]（可変）
  ├── 優先取引所: [自動最安値 / 指定]
  └── 実行後: [通知のみ / 自動実行]

通知:
  ├── Telegram通知: [ON/OFF]
  ├── Web通知: [ON/OFF]
  └── 通知先: [ユーザー / グループ]

ステータス: [有効 / 一時停止 / 無効]
```

#### 2.2.3 取引上限管理（可変式）

```
【グローバル設定】
  ├── 1回あたり取引上限: ¥[可変] （デフォルト: 100万）
  ├── 1日あたり取引上限: ¥[可変] （デフォルト: 1,000万）
  ├── 1週間あたり取引上限: ¥[可変] （デフォルト: 5,000万）
  ├── 1ヶ月あたり取引上限: ¥[可変]（デフォルト: 2億）
  └── 全て管理者が随時変更可能

【ユーザー別設定】
  ├── ユーザーごとに個別上限を設定可能
  ├── ロールに応じたデフォルト上限
  └── 一時的な上限引き上げ（承認制）

【取引所別設定】
  ├── 取引所ごとに1日の上限を設定可能
  └── リスク分散のため
```

## 3. 非機能要件

### 3.1 パフォーマンス

| 項目 | 要件 |
|------|------|
| レート更新頻度 | 10秒〜30秒ごと（設定可能） |
| 画面表示速度 | 初回ロード: 3秒以内 |
| 同時接続 | 最大20ユーザー |
| API呼び出し | 各取引所のRate Limit内で最大頻度 |
| データ保持期間 | レート履歴: 1年、取引履歴: 永久 |

### 3.2 セキュリティ

| 項目 | 要件 |
|------|------|
| 認証 | ログイン必須（メール+パスワード or OTP） |
| APIキー保管 | 暗号化保存（AES-256） |
| 通信 | HTTPS必須 |
| Telegram認証 | 登録済みTelegram IDのみ操作可能 |
| 操作ログ | 全操作を記録（改竄不可） |
| 2FA | 取引実行時に2段階認証（設定可能） |
| IP制限 | ホワイトリスト方式（設定可能） |

### 3.3 可用性

| 項目 | 要件 |
|------|------|
| 稼働時間 | 24/7 |
| デプロイ先 | Mac mini（初期）→ VPS移行可能 |
| バックアップ | 日次DB自動バックアップ |
| 障害通知 | Telegramに即時通知 |

## 4. 技術スタック（提案）

```
【バックエンド】
  ├── Runtime: Node.js (TypeScript)
  ├── Framework: Fastify or Express
  ├── DB: PostgreSQL（レート履歴・取引ログ）
  ├── Cache: Redis（リアルタイムレート・セッション）
  ├── Queue: Bull/BullMQ（自動取引ジョブ）
  └── Scheduler: node-cron（定期レート取得）

【フロントエンド】
  ├── Framework: Next.js (React)
  ├── UI: Tailwind CSS + shadcn/ui
  ├── チャート: TradingView Lightweight Charts
  ├── リアルタイム: WebSocket (Socket.io)
  └── テーマ: ダーク（デフォルト）

【Telegram Bot】
  ├── Library: grammy or node-telegram-bot-api
  └── バックエンドAPIと共有

【P2P取引所連携】
  ├── 各取引所のREST APIを直接呼び出し
  ├── APIがない取引所はスクレイピング（Playwright）
  └── 統一インターフェースでラップ

【インフラ】
  ├── 初期: Mac mini（Docker Compose）
  ├── 将来: VPS or AWS
  └── リバースプロキシ: Caddy or Nginx
```
